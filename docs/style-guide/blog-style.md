# 블로그 양식 표준 (STYLE.md)

> 이 문서는 **[AI의 정석]** 블로그의 모든 글이 따라야 하는 양식 표준이다.
> Validator(Stage 5)가 이 문서를 기준으로 14+8항목을 자동 검증하며, 위반 시 Reflection(최대 2회)이 발동된다.
> **이 프로젝트에서 가장 중요한 문서.**

---

## [A] 말투·문체 규칙 (반드시 준수)

### 기본 톤
- **존댓말** "~합니다 / ~입니다 / ~이죠 / ~됩니다" 기본
- 부드럽고 차분한 설명조, 독자와 대화하듯
- 문단은 **3~5줄**, 중간중간 단문으로 리듬 조절
- 과장된 감탄사/반말/영어 문장 금지

### 핵심 표현 기법
- **볼드 + 영문 병기**: `**에이전틱 RAG(Agentic RAG)**`, `**검색기(Retriever)**`
- **비유를 적극 사용** — 추상 개념은 반드시 일상적 비유로 설명
  - 예: "기존 RAG는 '수동적인 신입사원', 에이전틱 RAG는 '능동적인 수석 연구원'"
  - 예: "도구(Tools)는 에이전트가 활용할 수 있는 **무기들**입니다"
- **이모지는 포인트로만**: ✅ ❌ 🚀 ⚙️ 🗣️ 🤖 💡 📌 🔥 ⚠️ 📝 📚 (남발 금지)
- 섹션 전환 시 "~살펴보겠습니다 / 파헤쳐 보겠습니다" 류의 안내 문구

### 금지 사항
- "~요" 반말체 (예: "~거예요", "~해요") — 사용 금지
- 영어 문장 단독 사용 금지 (영어는 괄호 안 병기만)
- AI가 썼다는 티 (예: "분석해드리겠습니다~!" 같은 과한 친절함) 금지
- **"Co-Authored-By" 또는 AI 참여 표시 절대 포함 금지**

## [B] 구조·분량·깊이 규칙

### 전체 골격 (권장 구조 — 7~9개 대섹션)

```
[도입부: 인사 + 배경 + 이전 글 링크(있다면)]

목차
  1. 들어가며 (배경 + 왜 이 글을 읽어야 하는지)
  2. [개념] 이란? (정의 + 기존 방식과의 차이)
  3. 핵심 구성요소 (아키텍처 / 내부 구조)
  4. 특징 및 장단점
     - 장점
     - 단점 (반드시 4개 이하 카테고리로 통합)
     - 언제 쓰지 말아야 하는가 (필수 서브섹션)
  5. 주요 활용 분야 / 지원 범위
  6. [실습] 설치·구축
     - 설치 방법 (+ 선택 가이드 1줄 필수)
     - 주요 명령어·사용법
     - 첫 일주일 도입 로드맵 (필수 서브섹션)
     - Troubleshooting 표 (선택, SEO 강력)
  7. 한 눈에 비교 (비교표 중심)
  8. 마치며 (필수 — 마무리 멘트)

📝 정리 (불릿 5~7개)
(선택) JSON-LD TechArticle schema
```

**⚠️ 구조 원칙:**
- 대섹션은 **7~9개로 제한**. 10개 넘으면 독자가 피로해짐
- "등장 배경" 같은 섹션을 별도로 만들지 말고 **들어가며에 흡수**
- "실제 사용 사례"가 3~4줄로 짧으면 **독립 섹션 금지**, 장점 섹션에 병합
- "비용 플랜별 전략"은 **장점 섹션 안**에 넣기 (별도 섹션 X)

**⚠️ 포함하지 않는 섹션:**
- ❌ FAQ 섹션 (제외)
- ❌ 참고 자료 섹션 (제외)
- ❌ FAQ JSON-LD schema (제외)

### 🔴 중복 노출 절대 금지 (Anti-Duplication Rule)

같은 정보를 두 번 이상 노출하면 글이 20%씩 지루해집니다. 아래 항목은 **글 전체에서 1~2회만** 등장해야 합니다.

<tr style="background:#f5f5f5;"><th>항목</th><th>허용 노출 횟수</th><th>어디에</th></tr>
<tr><td>버전 번호 (예: v0.36.0)</td><td><b>1회</b></td><td>들어가며 또는 정의 섹션 택1</td></tr>
<tr><td>GitHub 스타/스펙 수치</td><td><b>1회</b></td><td>들어가며에만</td></tr>
<tr><td>주요 경고/함정 (예: "이름 충돌")</td><td><b>1회</b></td><td>실습 섹션에만 (단점에는 키워드로만 언급)</td></tr>
<tr><td>대표 절감률 수치 (예: "89%", "60~90%")</td><td><b>2회</b></td><td>들어가며 + 장점 섹션</td></tr>
<tr><td>대표 예시 명령어 (예: cargo test → 262 passed)</td><td><b>1회</b></td><td>작동원리 섹션에만</td></tr>
<tr><td>대표 비유 (예: "신입사원 vs 연구원")</td><td><b>1회</b></td><td>"기존 방식과의 차이" 서브섹션에만</td></tr>
</table>

**자가 검증**: 원고 완성 후 Ctrl+F로 버전 번호·스타수·대표 수치를 검색해 **3회 이상 등장하면 삭제**.

### 🎭 비유(Analogy) 사용 규칙 — 섹션당 1개

각 대섹션마다 **추상 개념을 일상 비유로 1개씩** 제시하면 독자 기억에 남습니다.

<table border="1" cellpadding="6" style="border-collapse: collapse;">
<tr style="background:#f5f5f5;"><th>섹션</th><th>비유 예시</th></tr>
<tr><td>정의</td><td>"우편물 필터링 비서", "프록시 = 중간 통역사"</td></tr>
<tr><td>아키텍처</td><td>"고속도로 톨게이트", "공장 컨베이어 벨트"</td></tr>
<tr><td>Tee/Proxy 같은 안전장치</td><td>"Tee는 블랙박스, Proxy는 녹음기"</td></tr>
<tr><td>트래킹/SQLite</td><td>"가계부 자동 기록"</td></tr>
<tr><td>마치며</td><td>"수동 → 자율"의 패러다임 전환</td></tr>
</table>

**규칙**: 같은 비유 재사용 금지. 섹션마다 **새로운 비유**를 하나씩 만들어야 신선함이 유지됨.

### 분량 가이드

| 레벨 | 총 분량 | 대섹션 수 | 용도 |
|------|---------|-----------|------|
| 표준 | 6,000~8,000자 | 7~9개 | 일반 개념 정리 |
| 장문 | 10,000~13,000자 | 9~11개 | 심화 가이드 |
| 초장문 | 15,000자+ | 11~13개 | 완벽 정복 시리즈 |

### 섹션 번호 체계
- 2단계까지 사용: `## 2. 제목` → `### 2-1. 소제목` → `### 2-2. 소제목`
- 3단계는 자제 (가독성 저하)

### 필수 포함 요소 (각 섹션마다)

1. **비교표 1개 이상** — **HTML `<table>` 태그 사용** (티스토리 호환)
2. **코드 블록 2~5개** — 반드시 `# 목적 주석` 포함
   ```bash
   # 대화형 모드 시작
   claude
   ```
3. **콜아웃 박스 1개 이상** — 아래 형식 활용
4. **비유 1개** (섹션의 핵심 개념을 일상 비유로 풀이 — 같은 비유 반복 금지)

### 단점 섹션 작성 규칙
- **나열식(7개 이상) 금지** — 반드시 **4개 이하 카테고리**로 통합
  - 좋은 예: ① 필터 정확도 리스크 ② 플랫폼 제약 ③ 통합 범위 ④ 역설적 비용
  - 나쁜 예: 1. Bash만 지원 2. Windows 제한 3. JSON 손상 4. chars/4 추정 5. 이름 충돌 6. 분포 불일치 7. 토큰 역설 (← 너무 잘게 쪼개짐)
- 각 카테고리 아래 **"언제 발생하는지" + "대응 방법"** 을 한 문단으로 서술

### 실습 섹션 필수 서브섹션

1. **설치 방법 선택 가이드 1줄** — 여러 설치법을 나열한 직후에 반드시 넣기
   - 예: "💡 macOS면 Homebrew, Rust 개발자면 Cargo, 빠른 체험이면 스크립트 권장"
2. **"첫 N일 도입 로드맵"** — 실습 섹션 끝에 체크리스트 형식으로
   - Day 1: 설치 + 초기 설정
   - Day 3: 첫 측정
   - Day 7: 실제 비용·세션 길이 비교
3. **Troubleshooting 표 (권장)** — 롱테일 SEO 키워드를 잡기 위한 Q&A형 매트릭스
   - 형식: `에러/상황 | 원인 | 해결 방법`

### 콜아웃 박스 템플릿

> 💡 **새로운 기능: [기능명]**
> [기능 설명]. 특히 [사용 시나리오]에서 유용하다.

> ⚠️ **주의**
> [주의사항]. [이유]. [대안].

> 🔥 **Pro Tip: [팁 제목]**
> [구체적 활용법].

> 📌 **참고**
> [부가 정보 / 가격 / 버전].

> 📝 **정리**
> - 포인트 1
> - 포인트 2

### 표(Table) 작성 패턴 — **HTML 테이블 필수**

> 🔥 **중요: 티스토리 호환을 위해 반드시 HTML `<table>` 태그를 사용합니다.**
> 마크다운 파이프 테이블(`| col | col |`)은 티스토리 스킨에 따라 CSS 스타일이 없어 **줄바꿈만 된 채 테두리 없이 깨져 보이는 현상**이 있습니다. HTML 테이블은 인라인 스타일을 줄 수 있고, 모든 스킨에서 안정적으로 렌더링됩니다.

**개념 비교표 (HTML):**
```html
<table style="border-collapse: collapse; width: 100%;">
  <thead>
    <tr style="background: #f5f5f5;">
      <th style="border: 1px solid #ddd; padding: 10px;">구분</th>
      <th style="border: 1px solid #ddd; padding: 10px;">A 방식</th>
      <th style="border: 1px solid #ddd; padding: 10px;">B 방식</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td style="border: 1px solid #ddd; padding: 10px;"><b>핵심 목적</b></td>
      <td style="border: 1px solid #ddd; padding: 10px;">...</td>
      <td style="border: 1px solid #ddd; padding: 10px;">...</td>
    </tr>
  </tbody>
</table>
```

**간소화 스타일 (짧은 표용):**
```html
<table border="1" cellpadding="8" style="border-collapse: collapse;">
  <tr><th>항목</th><th>값</th></tr>
  <tr><td>...</td><td>...</td></tr>
</table>
```

**하이라이트 행 (강조하고 싶은 행):**
```html
<tr style="background: #FFF8E1;">
  <td>...</td>
</tr>
```

### 마무리(마치며) 서사 — **필수 섹션**

> ⚠️ **"마치며" 섹션은 반드시 포함해야 합니다.** 사용자 본인 블로그의 고정 스타일이며, 이 섹션이 없으면 글이 미완성으로 보입니다.

**"현재의 한계 → 미래 전망 → 독자 실천 권유"** 3단 구조:
1. 지금까지의 기술/패러다임 요약 (1문단)
2. 비록 단점(X, Y)은 있지만, [분야]에서는 필수가 되고 있다 (1문단)
3. 오늘 실습한 내용을 바탕으로 독자가 직접 시도해볼 것 권유 (1문단, 🚀로 마무리)

## [C] 출력 형식

- **마크다운** (티스토리/벨로그/노션 호환)
- 맨 위에 제목 한 줄 + 카테고리 태그
- 두 번째 줄에 **목차 블록**
- 섹션 제목: `## 1. 제목`, 하위: `### 1-1. 소제목`
- 인라인 코드: `` `--flag` ``
- 코드 블록: 언어 지정 (```python, ```bash, ```typescript)
- 이미지 placeholder: `![대체텍스트](이미지_URL_또는_placeholder)`

## [D] 품질 체크리스트 (출력 전 자가 검증)

생성 후 아래를 반드시 확인 (Validator Stage 5에서 자동화):

- [ ] 존댓말 "~합니다/~입니다" 일관성 (반말·"~요" 혼입 없음)
- [ ] 비유가 최소 2개 이상 포함됨
- [ ] 각 대섹션에 표 최소 1개 (HTML `<table>`)
- [ ] 코드 블록에 목적 주석 포함
- [ ] 콜아웃 박스 3개 이상 (💡/⚠️/🔥/📌 중)
- [ ] 📝 정리 섹션 (5~7개 불릿)
- [ ] 마치며 섹션이 "한계 → 전망 → 실천" 3단 서사
- [ ] 목표 분량의 ±15% 이내
- [ ] 영문은 모두 괄호 병기 형태 (`**RAG(Retrieval-Augmented Generation)**`)
- [ ] AI가 썼다는 티 없음 (Co-Authored-By, "분석해드렸습니다~" 등 없음)
- [ ] FAQ 섹션 없음 (제외됨)
- [ ] 참고 자료 섹션 없음 (제외됨)
- [ ] 카테고리 태그 정확히 10개
- [ ] 비교·스펙 표는 모두 HTML `<table>` 태그 사용

## [E] 참고자료 수집·활용 전략

### 수집 규칙 — 두 가지 경로

#### 경로 A: 사용자가 참고자료를 직접 제공한 경우

1. **제공된 자료가 최우선 1차 소스** — 반드시 모두 WebFetch로 읽고 내용 파악
2. 제공 자료가 **12개 미만이거나 특정 버킷이 비어 있으면** WebSearch로 **보충**하여 총 12~20개 채움
3. 제공 자료가 이미 충분하면(12개 이상, 버킷 균형) 추가 검색 **최소화**, 하지만 **최신 코드 검증을 위한 공식문서 확인은 반드시 수행**
4. 사용자가 준 자료의 **발행일이 1년 이상 지난 경우** 최신 공식문서로 교차 검증 필수

#### 경로 B: 사용자가 자료를 주지 않은 경우

WebSearch로 전량 직접 수집. 아래 6버킷 규칙 그대로 적용.

### 6개 버킷 수집 기준

아래 6개 버킷에서 **각 2~4개씩, 총 12~20개**를 모읍니다.

| 버킷 | 예시 | 최소 개수 |
|------|------|-----------|
| 🏛️ **공식 문서** | Anthropic, OpenAI, LangChain, LlamaIndex 공식 docs | 2~3 |
| 📄 **논문/기술 리포트** | arXiv, Google Research, Meta AI 블로그 | 2~3 |
| 💻 **GitHub 저장소** | 대표 구현체, star 1k+ 리포 | 2~3 |
| 🎥 **영상 강의** | YouTube, DeepLearning.AI, 공식 컨퍼런스 발표 | 2~3 |
| 🌐 **영문 기술블로그** | Medium, Towards Data Science, 공식 엔지니어링 블로그 | 2~3 |
| 🇰🇷 **한글 블로그/문서** | 기술 블로그, 네이버 D2, 카카오 기술블로그 | 2~3 |

### 배치 방식
- 본문 중간 인용 시: "[Anthropic 공식 문서](url)에 따르면..." 자연스럽게 삽입
- 링크는 반드시 **실제 URL**로. 추측·날조 금지. 확신 없으면 "공식 문서" 텍스트만 남기기

### ⚠️ 중복 회피 규칙 (핵심)

참고자료의 내용을 **절대 그대로 복제하지 않습니다.** 개념과 구조만 차용하고, 표현은 완전히 새로 씁니다.

**실습 코드 작성 시 필수 변형:**
1. **도메인 예시 변경**: 참고가 "사내 PDF 검색"이면 → "의료 논문 검색", "법률 판례 검색" 등으로 교체
2. **변수·함수명 변경**: `vectorstore` → `knowledge_base`, `retriever` → `doc_finder` 식으로 의미 유지하되 명칭 차별화
3. **프레임워크 변주**: 참고가 LangChain이면 → LangGraph 또는 LlamaIndex 구현도 함께 제시
4. **코드 주석 재작성**: 참고 주석을 복붙하지 말고 자신의 언어로 재기술
5. **구조 변경**: 클래스 기반 → 함수형, 혹은 그 반대. 프롬프트 템플릿 문구도 다르게
6. **유사도 체크**: 참고 코드와 **30% 이상 일치 라인이 없도록** 자가 검증
7. **실제 검증**: WebSearch로 해당 라이브러리의 **최신 API 시그니처**를 확인해 deprecated 함수 회피

**본문 서술 시:**
- 참고 블로그의 문장·비유를 그대로 쓰지 말고, 본인 비유로 재창조
- 동일 개념이라도 **다른 각도**로 설명 (예: 참고가 "프로세스" 관점이면 → "데이터 흐름" 관점)
- 참고에 없는 **본인만의 인사이트 1~2개** 반드시 추가 (예: "실무에서 주의할 함정", "비용 최적화 팁")

## [F] SEO + AEO + GEO 3대 최적화 규칙

### F-1. SEO 최적화 (검색엔진 상위 노출)

#### 제목(Title) 규칙
- **60자 이내**, 주 키워드를 **앞쪽 30자** 안에 배치
- `[카테고리]` 태그 포함, 핵심 내용을 한 줄로 압축
- 숫자·연도 포함 시 CTR 상승 (예: "2026년 완벽 가이드", "5가지 핵심 패턴")
- 좋은 예: `[AI/LLM] 에이전틱 RAG(Agentic RAG) 완벽 가이드 2026 (정의·구조·실습)`

#### 카테고리 태그 규칙 — **정확히 10개 필수**

<table border="1" cellpadding="6" style="border-collapse: collapse;">
<tr style="background:#f5f5f5;"><th>종류</th><th>개수</th><th>예시</th></tr>
<tr><td>주 키워드(한글/영문)</td><td>2개</td><td>#에이전틱RAG #AgenticRAG</td></tr>
<tr><td>카테고리/분야</td><td>2개</td><td>#AI검색 #LLM응용</td></tr>
<tr><td>핵심 기술/개념</td><td>2개</td><td>#RAG파이프라인 #벡터검색</td></tr>
<tr><td>관련 제품/플랫폼</td><td>2개</td><td>#LangChain #LlamaIndex</td></tr>
<tr><td>트렌드/맥락 키워드</td><td>2개</td><td>#AI에이전트2026 #2026</td></tr>
</table>

형식: `#태그1 #태그2 #태그3 ... #태그10` (공백 구분, 정확히 10개)

#### 메타 디스크립션 (도입부 첫 문단)
- **첫 100~150자 안에 주 키워드 2~3회** 자연스럽게 반복
- 독자의 검색 의도(pain point)를 거울처럼 반영
- 이 글이 해결해줄 것을 명확히 약속

#### 헤딩(Heading) 키워드 전략
- **H2(`##`)**: 주 키워드 + 의도 키워드 조합
  - 예: `## 2. 에이전틱 RAG(Agentic RAG)란? — 정의와 작동 원리`
- **H3(`###`)**: 롱테일 키워드 삽입
  - 예: `### 4-2. 에이전틱 RAG 아키텍처 패턴 3가지 비교`
- H2/H3 전체 제목에 주 키워드 또는 변형이 **최소 40% 이상** 포함

#### LSI(의미 연관) 키워드 자연 삽입
- 주 키워드 외 연관어를 **본문 전반에 분산**, 키워드 밀도 **1~2%** 유지

#### Article Schema (TechArticle JSON-LD)

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "TechArticle",
  "headline": "...",
  "description": "...",
  "datePublished": "YYYY-MM-DD",
  "dateModified": "YYYY-MM-DD",
  "author": {
    "@type": "Person",
    "name": "Jaylen Han",
    "url": "https://jaylenhan.tistory.com"
  },
  "publisher": {
    "@type": "Organization",
    "name": "AI의 정석"
  },
  "keywords": "주키워드, LSI키워드1, LSI키워드2, ...",
  "articleSection": "AI/LLM",
  "wordCount": 8000
}
</script>
```

#### 이미지 alt & 파일명
- 모든 이미지에 `alt="키워드 포함 설명"` 필수
- 파일명: `keyword-slug.png` (공백/한글 X)

#### 독자 체류시간 최적화
- 글 초반에 **"이 글을 다 읽으면 얻게 되는 것 3가지"** 불릿 제시
- 섹션 사이에 예고 문구로 이탈 방지
- 내부 앵커·목차로 빠른 네비게이션 제공

### F-2. AEO 최적화 (AI 답변엔진 인용)

**AEO(Answer Engine Optimization)** — Perplexity, Bing Chat, Google AI Overview가 블로그 내용을 **인용 소스로 채택**하도록 최적화하는 전략.

#### 1. 정의문 패턴 (Definition Statement)

각 대섹션의 **첫 문단**에 "~란 ~하는 ~이다" 형태의 명확한 정의문을 삽입한다.

```markdown
## 2. 에이전틱 RAG(Agentic RAG)란?

**에이전틱 RAG(Agentic RAG)란** 기존 RAG 파이프라인에
**자율 판단 에이전트**를 결합하여, 검색 전략을 스스로 계획·실행·평가하는
차세대 검색 증강 생성 기법입니다.
```

**규칙:**
- 각 대섹션 H2 바로 아래 첫 문단에 1개
- "~란"으로 시작, "~이다/~합니다"로 종결
- 1~3문장, 100자 이내 권장
- AI가 "X란 무엇인가?" 질문에 이 문장을 직접 인용할 수 있어야 함

#### 2. 핵심 요약 박스 (Summary Box)

각 섹션의 핵심을 1~2문장으로 압축한 박스를 섹션 상단에 배치.

```markdown
> 💡 **핵심**: 에이전틱 RAG는 쿼리 분석 → 다중 소스 라우팅 → 결과 평가 →
> 필요 시 재검색이라는 4단계 자율 루프를 통해 기존 RAG 대비 정확도를 40% 향상시킵니다.
```

**규칙:**
- 각 대섹션에 0~1개 (핵심 섹션만)
- 정량 수치 포함 권장

#### 3. 비교표 Q&A 구조

비교가 필요한 곳에 "질문 → 표 답변" 패턴을 사용.

```markdown
### 기존 RAG vs 에이전틱 RAG, 무엇이 다른가?

<table>
<tr><th>비교 항목</th><th>기존 RAG</th><th>에이전틱 RAG</th></tr>
<tr><td>검색 전략</td><td>고정 파이프라인</td><td>자율 판단</td></tr>
</table>
```

AI 답변 엔진이 "A와 B의 차이점은?"이라는 질문에 이 표를 직접 인용.

#### 4. HowTo Schema (실습 섹션)

설치/구축 섹션에 JSON-LD HowTo 구조화 데이터를 추가.

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "HowTo",
  "name": "에이전틱 RAG 구축하기",
  "step": [
    {
      "@type": "HowToStep",
      "name": "LangChain 설치",
      "text": "pip install langchain langchain-openai"
    }
  ]
}
</script>
```

### F-3. GEO 최적화 (생성형 AI 학습 데이터 채택)

**GEO(Generative Engine Optimization)** — ChatGPT, Claude, Gemini가 블로그 내용을 **학습 데이터 또는 참조 소스로 채택**하도록 최적화하는 전략.

#### 1. 독창적 관점 섹션 (Unique Insight)

다른 블로그에 없는 **고유 분석, 실험 결과, 경험담**을 1개 이상 포함.

```markdown
### 직접 테스트한 결과

필자가 동일한 100개 질의셋으로 비교 테스트한 결과:

<table>
<tr><th>지표</th><th>기존 RAG</th><th>에이전틱 RAG</th><th>개선율</th></tr>
<tr><td>정확도</td><td>72.3%</td><td>89.1%</td><td>+23.3%</td></tr>
</table>
```

**규칙:**
- 매 글에 1개 이상 (Could: 2~3개)
- 정량 수치 필수 포함
- AI가 학습할 때 "새로운 정보"로 인식 → 인용 확률 상승

#### 2. 정량적 수치 (Quantitative Data)

```
✅ 좋은 예: "에이전틱 RAG는 기존 RAG 대비 정확도를 23.3% 향상시킨다"
❌ 나쁜 예: "에이전틱 RAG는 기존 RAG보다 훨씬 정확하다"
```

AI는 구체적 수치가 있는 문장을 "근거 있는 정보"로 판단하여 인용 확률이 높아짐.

#### 3. 인용 가능한 한 줄 정의 (Quotable Definition)

핵심 개념마다 **볼드 처리 + 독립 문단**으로 한 줄 정의문을 배치.

```markdown
**에이전틱 RAG(Agentic RAG)는 "검색을 지시하는 AI"에서 "검색을 스스로 계획하는 AI"로의 패러다임 전환이다.**
```

**규칙:**
- 글 전체에서 2~3개
- 볼드 + 독립 문단 (전후 빈 줄)
- AI가 답변에서 그대로 인용할 수 있는 완결된 문장

#### 4. E-E-A-T 신호 강화

| 신호 | 적용 방법 | 예시 |
|------|----------|------|
| **Experience** | 실제 사용 경험 포함 | "3개월간 프로덕션에서 운영한 결과..." |
| **Expertise** | 기술적 깊이 (코드, 아키텍처) | 동작 원리 + 코드 + 다이어그램 |
| **Authoritativeness** | 공식문서/논문 기반 서술 | "[공식 문서에 따르면...]" |
| **Trustworthiness** | 단점/한계 솔직히 기술 | "그러나 이 방식에는 세 가지 한계가 있습니다" |

## [I] 최종 출력 전 자가 검증 (반드시 수행)

본문을 제시하기 직전에 아래를 스스로 확인. 하나라도 미충족이면 수정 후 출력.

### 내용 품질

- [ ] `[A] 말투` 체크리스트 통과
- [ ] `[B] 구조` 체크리스트 통과 (HTML 표·콜아웃·정리 모두 존재, **FAQ·참고자료 제외**)
- [ ] `[E] 중복 회피` — 코드/문장 참고자료와 30% 이상 일치 없음

### SEO 체크 (8항목)

- [ ] 제목 60자 이내, 주 키워드 앞쪽 30자
- [ ] 태그 정확히 10개
- [ ] 메타 디스크립션 100~150자, 주 키워드 2~3회
- [ ] H2/H3 키워드 40%+
- [ ] 키워드 밀도 1~2%
- [ ] 이미지 alt 태그 전수
- [ ] JSON-LD TechArticle 존재
- [ ] LSI 키워드 10개 이상 자연 삽입

### AEO 체크 (4항목)

- [ ] 정의문 패턴 대섹션별 1개
- [ ] 핵심 요약 박스 3개 이상
- [ ] 비교표 Q&A 구조 1개 이상
- [ ] HowTo Schema (실습 섹션 있을 때)

### GEO 체크 (4항목)

- [ ] 독창적 관점 섹션 1개 이상
- [ ] 정량적 수치 5개 이상
- [ ] 인용 가능 정의문 2~3개
- [ ] E-E-A-T 4가지 신호 중 3가지 이상

### 사용자 요구사항

- [ ] 목차가 "1. 들어가며"로 시작
- [ ] 대섹션 **7~9개** (10개 넘지 않음)
- [ ] "한 눈에 비교" 섹션 + **"마치며" 섹션(필수)** 으로 종료
- [ ] **FAQ 섹션 없음**, **참고자료 섹션 없음** (제외됨)
- [ ] 카테고리 태그 **정확히 10개**
- [ ] 비교·스펙 표는 모두 **HTML `<table>` 태그 사용**
- [ ] 실습 코드는 웹 검색 기반의 최신·자체작성 버전
- [ ] AI 작성 티 없음

### 중복·밀도 검증

- [ ] 버전 번호/스타수 **3회 미만** 등장
- [ ] 대표 절감률 수치 **3회 미만** 등장
- [ ] 같은 경고 문구 **2회 미만** 반복
- [ ] 같은 비유 **재사용 없음**

### 구조 품질

- [ ] 단점 섹션이 **4개 이하 카테고리**로 통합됨
- [ ] 실습 섹션에 **설치 선택 가이드 1줄** + **첫 N일 로드맵** 포함
- [ ] 각 대섹션에 **비유 1개** 이상
- [ ] "등장 배경" 같은 중복 섹션 **삭제됨** (들어가며에 흡수)
