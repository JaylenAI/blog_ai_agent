# Phase 1 · 문제 정의 및 기획 의도

> 💡 부록 E Phase 1 "왜 이걸 만드는가?"를 명확히 해야 개발 도중 방향을 잃지 않는다.

---

## 1-1. 문제 정의서 (Problem Statement)

### 🔴 현재 상황 (As-Is)

`jaylenhan.tistory.com` "AI의 정석" 블로그를 운영하면서 마주하는 실제 작업 흐름:

```
주제 결정
  ↓
자료조사 — 공식 문서, 블로그, GitHub, Stack Overflow 검색 (60-90분)
  ↓
아웃라인 잡기 — 사용자 양식 9개 섹션에 맞춰 구조 설계 (20-30분)
  ↓
본문 초안 — 각 섹션 작성, 격식체 톤 유지, 코드 예시 작성 (120-180분)
  ↓
다이어그램 5장 — Excalidraw/Draw.io로 직접 그리기 (60-90분)
  ↓
썸네일 만들기 — Figma 또는 캔바 (30-45분)
  ↓
양식 점검 — H2 별표 굵게, 인용구 이탤릭, 박스 이모지 등 (15-20분)
  ↓
SEO 메타 + JSON-LD 작성 (15-20분)
  ↓
Tistory 에디터에 마크다운 → HTML 변환해서 붙여넣기, 이미지 업로드 (30-45분)
  ↓
발행
```

→ **편당 4-8시간 (평균 5-6시간)**. 주 1편 페이스 유지가 어렵고, 매주 2-3편 발행은 사실상 불가능.

### 🔴 페인 포인트 (Pain Points)

각 단계의 구체적 어려움:

| # | 단계 | Pain Point | 영향도 |
|---|------|-----------|------|
| 1 | 자료조사 | 검색 엔진 → 클릭 → 스크롤 → 핵심만 추출 작업이 반복적. 30개 탭이 열려도 정작 활용은 5개 | 🔥 매우 큼 |
| 2 | 아웃라인 | 양식 9개 섹션이 정해져 있는데, 매번 처음부터 짜는 느낌. 기계적 작업 | 🟡 중간 |
| 3 | 본문 작성 | 격식체 유지 + 코드 예시 + ❌/✅ 대비 등 양식 디테일이 흔들림 | 🔥 매우 큼 |
| 4 | 다이어그램 | 그리기 도구 따로, 색상/폰트 일관성 유지 어려움, 한 번 그리고 수정하려면 또 도구 열어야 함 | 🔥 매우 큼 |
| 5 | 썸네일 | 매번 처음부터 디자인. 카테고리별 통일된 브랜드 없음 | 🟡 중간 |
| 6 | 양식 점검 | H2 별표 누락, 인용구 이탤릭 안 함 같은 미세한 위반 발견이 어려움 | 🟢 작음 |
| 7 | SEO/JSON-LD | 글마다 처음부터 작성. 매번 빼먹는 필드 있음 | 🟡 중간 |
| 8 | Tistory 입력 | 에디터가 마크다운 미지원. 코드 블록/표/이미지 일일이 옮기기 | 🔥 매우 큼 |

### 🟢 원하는 상태 (To-Be)

```
주제 한 줄 결정
  ↓
Claude Code에 "X 블로그 만들어줘" 입력
  ↓
[자동] 자료조사 5-8 검색 병렬, 5-10개 출처 정리 (2-3분)
  ↓
[자동] 아웃라인 생성 (양식 9 섹션, 각 섹션 핵심 주장 + 출처) (1-2분)
  ↓
🛑 Gate 1: 사용자가 아웃라인 검토 (5분)
  ↓
[자동, 병렬] 각 H2 섹션 본문 작성 (4-6 동시) (1-2분)
  ↓
[자동, 병렬] Mermaid 다이어그램 4-5장 + HTML 썸네일 + SVG 일러스트 (1-2분)
  ↓
[자동] 통합 + Critic + Reflection 2회 (양식 100% 검증) (1분)
  ↓
🛑 Gate 2: 사용자 최종 검수 (10-15분)
  ↓
[자동] Tistory Playwright 자동 입력 (3-5분, 카카오 로그인만 수동)
  ↓
발행
```

→ **자동 5-8분 + 검수 15-20분 = 편당 20-30분**. 주 2-3편 발행 가능.

### 🎯 타겟 사용자

**1차 (즉시)**: Jaylen H (본인)
- 블로그 `jaylenhan.tistory.com` 운영
- AI Engineer를 목표로 학습 중인 대학생
- 주 1편 페이스 유지 어려움 → 주 2-3편으로 확장하고 싶음

**2차 (검증 후 확장)**: 같은 패턴의 한국 개발자 블로거
- Tistory / Velog 운영
- 정형화된 양식을 가지고 있음
- 자동화에 거부감 없는 개발자

**3차 (장기, [[15-roadmap|Phase 15 로드맵]] 참조)**: 오픈소스화
- 자신만의 STYLE.md를 정의 가능한 사용자
- 다른 플랫폼(Velog/Hashnode) 어댑터 기여 가능자

### 🏁 성공 기준

MVP 성공 기준:
- [ ] 주제 한 줄 → Tistory 게시까지 30분 이내 (검수 포함)
- [ ] 양식 STYLE.md 100% 준수 (Critic 검증 통과)
- [ ] 편당 비용 $0 (Claude OAuth 구독 외)
- [ ] 본인이 만족할 품질의 글 1편 실제 게시
- [ ] 블로그 1주일 활동으로 평균 글 수 3편 이상 달성

확장 성공 기준:
- [ ] 30편 이상 자동 생성 후 양식 일관성 유지
- [ ] 잘못된 사실 자동 검출률 80% 이상 (Critic Fact-check)
- [ ] 사용자(본인) 만족도 4.0/5.0 이상 (재작성 비율 20% 이하)

## 1-2. 기획 의도 — "왜?" 세 가지 답변

부록 E §1-2의 3가지 핵심 질문에 답한다.

### Q1. 왜 이 문제?

❌ "재미있을 것 같아서"

✅ **현실적이고 정량화 가능한 이유**:
- 본인 블로그 게시 빈도 = 월 4-5편 (목표 월 10-12편 대비 50% 미달)
- 편당 작업 시간 5-6시간 = 월 25시간 = 학기 중 학업과 병행 불가
- 자료조사 단계가 전체의 30-40% 차지 → AI로 가장 큰 ROI 발생 지점

### Q2. 왜 이 방식?

❌ "AI가 트렌드라서"

✅ **기술 선택에 근거**:
- LLM은 자료조사 + 양식 적용 + HTML 변환에 가장 강점
- 사람은 "무엇을 쓸지 결정" + "마지막 검수"에 가장 강점
- → 둘의 분업이 자연스러움 = Multi-stage 파이프라인 + Gate 2개 패턴

근거 자료:
- Anthropic Research (2025): 멀티 에이전트가 복잡 연구 작업에서 90.2% 성능 향상
- 사용자 1차 검증: Prometheus 블로그 글 1편을 AI로 생성해본 결과 (첨부 자료, 2099줄)
- 시장 검증: Jasper, Copy.ai 같은 영어 자동화 도구는 이미 사용자 십만 명대

### Q3. 왜 지금?

❌ "졸업 전에 해야 해서"

✅ **지금이어야 하는 객관적 이유**:
- **Claude Agent SDK 정식 출시** (2025년 9월) — Skills 시스템 안정화
- **Claude Code 구독료가 낮음** ($20/월) — API 직접 사용 대비 1/10
- **MCP 생태계 성숙** (Context7, grep.app 등) — 자료조사 인프라 무료
- **Tistory Open API 종료** (2024.02) → Playwright 자동화가 유일한 길이지만, Playwright도 안정화됨
- **사용자 운영 데이터 누적** — jaylenhan.tistory.com 100+ 글로 양식 패턴 추출 가능

→ 6개월 전에는 불가능, 6개월 후에는 늦음. **2026년 5월이 정확한 타이밍**.

이 프로젝트는 **사이드 프로젝트 + 포트폴리오용**. 그 유형에 따른 강조점:

| 항목 | 이 프로젝트의 적용 |
|------|-------------|
